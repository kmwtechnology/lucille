The following requirements govern Lucille's architecture. They evolved through production deployments and customer engagements, and are presented here to explain *why* the system is designed the way it is.

**1. Parallelism.** The three tasks of retrieving data, processing it, and sending it to the search engine should be handled by separate components that run alongside each other. In Lucille, these components are the Connector, Worker, and Indexer.

**2. Scalability.** Lucille should provide a way to dedicate more system resources to accommodate heavier load. This is accomplished by allowing multiple Worker and Indexer instances to run concurrently, and by allowing the number of Workers and Indexers to be increased while ingestion is in progress.

**3. Flexible deployment.** Lucille should provide a simple deployment model where all components run as threads inside a single Java process. But it should also support a distributed deployment model where each component instance runs as its own dedicated Java process, communicating through a distributed messaging system. Where the separation between Worker and Indexer is not required, Lucille should support a hybrid model where Worker/Indexer pairs run in their own processes.

**4. Deployment-independent behavior.** Lucille should function the same way and use the same code paths regardless of which deployment model is used. This is achieved by having all components use clearly-defined APIs for inter-component communication. In the distributed model, these APIs are backed by Kafka; in the local model, they are backed by in-memory queues. In switching deployment models, everything remains the same except the communication implementation.

**5. Child documents.** Lucille should support "child" Documents created inside the processing Pipeline. While most work is generated by the Connector, a Worker can also generate new work — for example, splitting a zip file into its component files, each of which flows through downstream Stages and is indexed as its own record.

**6. Batch and streaming execution.** Lucille should support a batch execution model (finite data, clear completion) and a streaming execution model (unbounded data, continuous processing).

**7. Completion detection.** In the batch execution model, Lucille must provide a clear indication of when work is complete. This is complex in a distributed framework with multiple Indexer instances running separately from the Connector, each asynchronously completing items of work, and where new work can be generated inside Workers. Lucille solves this with a distributed accounting system: the Publisher tracks work sent into the system, Workers notify the Publisher when additional work is generated, and Indexers notify the Publisher when work is completed.

**8. Sequential composition.** Lucille must provide a way of composing multiple batch ingests into a sequence, with strict ordering — one ingest cannot start until all work from the previous ingest is complete.

**9. Streaming without a Connector.** In streaming mode, Lucille should provide a way to run without a Connector, where an external component places work on a Kafka topic and Workers retrieve it from that topic.

**10. Observability.** Each component should report metrics about its performance continuously during ingestion, with summaries at the end. Components should have a way of issuing a "heartbeat" to indicate they are still alive.

**11. Per-document error handling.** Lucille should handle exceptions during Document processing without stopping the ingest. It should detect and skip "poison pill" Documents that repeatedly cause a Worker process to crash, routing them to a Dead Letter Queue.

**12. Crash resilience.** Work should not be lost if a distributed Worker or Indexer crashes during an ingest. Uncompleted or in-progress work should be resumed when the component is restarted, or picked up by another available instance.

**13. Flexible Document model.** Lucille should provide a Document model that makes it easy and concise to perform common manipulations of Document data.

**14. Extensibility.** Lucille should provide a modular project structure in which new plugins can be defined without impacting the core project. There should be a clear framework for new Stages, Connectors, and Indexers.

**15. Configuration-driven operation.** All aspects of an ingest — connection information, pipeline definitions — should be specified in a readable configuration file. Changes should be possible without rebuilding the project. Configuration should be reusable across files and validatable without execution.

**16. Operation ordering.** Sequences of creates, updates, and deletes for the same document must preserve their order even when running in distributed mode with multiple Workers and Indexers.

**17. Batch indexing.** Indexers should send documents in configurable batches with size and timeout. Multiple search backends supported with and without security.

**18. Source coverage.** Connectors for databases, filesystems, CSV/XML/JSON, and search engines.

**19. Connector lifecycle.** Pre-execution, execution, and post-execution phases.

**20. Document collapsing.** Consecutive same-ID documents can be merged before processing.

**21. Built-in stages.** Common processing tasks available out of the box.

**22. Conditional execution.** Stages execute only when document meets configured criteria.

**23. Testability.** Full run history capturable in test mode for assertion by test code. Recording capability enabled only in test context.

**24. Inspectable data in flight.** Documents on Kafka topics should be human-readable.
